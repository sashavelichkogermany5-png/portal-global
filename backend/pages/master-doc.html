1. About the Project

1.1. Mission, Vision, Values

Mission
To give teams and operators a single operational hub for workspaces, projects, access control, and integrations, so they can run operations from one place with less fragmentation and better visibility.

Vision
To build a practical, API-first operational platform for modern businesses that need structured collaboration, automation, and tenant-aware access control without stitching together too many separate tools.

Values

  • Security first: tenant-aware architecture, least-privilege access, and clear operational controls.
  • Practical automation: APIs and integrations should reduce manual work, not add complexity.
  • Product honesty: public-facing pages should reflect what is live now, what is in testing, and what is still planned.
  • Customer feedback as product input: priorities should be shaped by real usage, not only by roadmap theory.

1.2. Problem and Solution

Problem
Many teams run operations across disconnected tools for tasks, communication, knowledge, permissions, and reporting. This creates duplicated data, broken context, access-control risks, and slow onboarding.

Solution
PORTAL Global is a tenant-aware B2B operations platform that brings together workspace structure, role management, project flows, dashboards, and integration endpoints in one system. The current product should be described as a live public prototype / early product release rather than a fully mature enterprise suite.

1.3. Core Capabilities

  • Workspaces and project areas
  • User roles and tenant-aware access
  • Dashboard and health/status surfaces
  • REST-style API endpoints for integrations
  • Foundations for automation and future agent workflows
  • Security-oriented product direction with auditability and controlled access

1.4. Current Status (Project Snapshot)

  • Product stage: live public build / active prototype under continued development.
  • Current runtime: unified application server handling landing pages, login, app shell, and API from a single deployment path.
  • What is real today: public access, app shell, authentication flow, tenant-oriented structure, dashboard surfaces, and core API routes.
  • What should be treated as in progress: live billing completion, polished documentation, advanced automations, Telegram flows, and some marketing/intro surfaces.
  • Recommended public wording: "PORTAL Global is live and available for testing and early adoption. Some modules are production-ready, while others are still being finalized."

1.5. Current Limitations

  • Current deployment is still relatively monolithic.
  • Not every planned module is consistently available in the public build.
  • Rate limiting exists, but plan-based differentiation may still be incomplete.
  • Billing should only be marketed as live after full end-to-end verification in production.
  • Legal templates still require final operator details and, ideally, legal review.
  • Customer-facing content should avoid implying that every roadmap module already exists.

The gateway / edge layer should handle

  • Rate limiting by IP, token, and user context
  • Authentication and request validation
  • Routing and header policy
  • Request logging and metrics
  • Separation of public traffic from internal/admin traffic

3.2. Multi-Tenancy and Data Isolation

Recommended product wording

  • Tenant isolation is a core design requirement.
  • Every request that touches protected data must resolve tenant context explicitly.
  • Authorization must verify both identity and tenant/resource ownership.
  • Implementation details may evolve over time without changing the promise of tenant isolation.

Example middleware concept

// Resolve tenant context from session, token, or host mapping
// Validate membership / role
// Apply tenant-aware query filters before reading or writing data

Practical note
Avoid claiming a specific database strategy publicly unless that strategy is stable and verified in production. Promise the outcome (tenant isolation), not an implementation detail that may still change.

3.3. API Versioning and Documentation

  • Use explicit versioning where appropriate, such as /api/v1/....
  • Publish API documentation only for endpoints that are actually available in the deployed environment.
  • Mark experimental endpoints clearly.
  • Provide a changelog for breaking changes.

3.4. Rate Limiting

Endpoint class Indicative limit Key
Public read endpoints 1000 req/min IP + token where available
Authenticated write endpoints 100 req/min User / token
Login 5 attempts/min IP
Registration 3 attempts/min IP
Dashboards / charts / overview 60 req/min User / session

Client rules for handling 429 (Too Many Requests)

  • Use exponential backoff.
  • Do not retry forever; maximum retry attempts: {MAX_RETRY_ATTEMPTS}.
  • Stop automated retries on 401/403 for identity-sensitive endpoints until user action occurs.
  • Prevent request storms in app boot flows and auth checks.

3.5. Security Checklist

  • TLS enabled on all public domains and subdomains
  • Security headers configured: CSP, X-Content-Type-Options, Referrer-Policy, and frame restrictions where applicable
  • Brute-force protection on login and account-creation routes
  • Audit logging for critical actions
  • Dependency scanning and update process
  • Secrets kept out of client bundles and public repos
  • Tenant ownership checks enforced in backend logic
  • Production claims limited to controls that are actually enabled

3.6. Backups and Restore

Target approach

  • Database backups with scheduled snapshots / dumps
  • File backups where user-generated files or uploads exist
  • Restricted access to backup material
  • Regular restore tests

3.7. Monitoring and Alerting

  • Availability monitoring for landing, login, app, and API routes
  • Error-rate and latency tracking
  • Structured logs with alertable signals for 5xx spikes, auth failures, and rate-limit storms
  • Notifications through email, Telegram, or another incident channel

  • SaaS subscription as the primary model
  • Value metrics may include users, tenants, integrations, usage volume, or support level
  • Additional revenue may come from enterprise onboarding, customization, implementation, or premium support

Value drivers

  • Centralized operational visibility
  • Controlled access and tenant-aware structure
  • Integration readiness and workflow simplification

4.3. Payment Provider

Practical recommendation: start with Stripe for subscription handling. Describe billing publicly as "available" only after a complete production payment flow, invoice handling, and failure handling have been tested end to end.

4.4. Subscription Flow

  • User selects a plan.
  • User creates an account or continues from a registration flow.
  • Payment details are collected through the payment provider.
  • On successful activation, the customer gets an admin account and initial workspace / tenant context.
  • Confirmation email and onboarding materials are sent.

4.5. Partner Program

  • Referral-based commercial partnerships
  • Optional revenue share for referred customers
  • Separate partner terms and compliance review recommended before launch
