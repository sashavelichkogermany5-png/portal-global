1. О проекте

1.1. Миссия, видение, ценности

Миссия
Предоставить компаниям единый операционный хаб, который централизует управление проектами, доступом и интеграциями, обеспечивая бесшовную совместную работу и полный контроль.

Видение
Стать стандартной операционной системой для B2B-компаний по всему миру, где каждая команда, проект и интеграция живут в безопасной среде с API-first подходом.

Ценности

  • Безопасность прежде всего: мультитенантная архитектура с изоляцией данных и встроенным соответствием DSGVO.
  • API-управляемость: каждая функция доступна через API для автоматизации и интеграций.
  • Клиентоцентричность: развитие продукта на основе обратной связи.
  • Прозрачность: понятные цены, открытая коммуникация и ясные правила обработки данных.

1.2. Проблема и решение

Проблема
Компании используют множество разрозненных инструментов (Jira, Trello, Slack, базы знаний, кастомные решения), что ведет к потере контекста, дублированию данных, сложностям с разграничением доступа и высокой стоимости поддержки интеграций.

Решение
PORTAL Global - B2B-платформа, объединяющая управление проектами, ролями и тенантами, аналитическую панель и REST API для интеграций. Мультитенантная архитектура позволяет безопасно работать с несколькими организациями из одного аккаунта. Встроенные rate limiting и версионирование API повышают стабильность при росте нагрузки.

1.3. Ключевые возможности

  • Проекты и рабочие пространства
  • Роли и тенанты (multi-tenancy)
  • Панель управления (dashboard) с метриками
  • REST API первого класса
  • Интеграции с внешними сервисами через API
  • Enterprise-ready принципы (изоляция данных, аудит, SSO - по плану)

1.4. Текущий статус (Project Snapshot)

  • Статус разработки: рабочий прототип, доступный для внешнего тестирования.
  • Режим: community mode (без автопилота, с ограничениями).
  • Архитектура прототипа: единый сервер на порту :3000 (landing + login + app + REST API).
  • Что уже реализовано: проекты, роли, тенанты, dashboard, rate limiting, REST API.
  • Публичный запуск: 15 марта 2026 (обновлено 01.03.2026). Причина: завершение billing + security hardening + финальная проверка legal страниц.

1.5. Ограничения текущей версии

  • Единый сервер (монолит) без разделения статики и API.
  • Отсутствие автопилота и автоматического масштабирования.
  • Rate limit включен, но еще не дифференцирован по тарифам.
  • Нет интеграции с платежными системами (в процессе).
  • Юридические документы в шаблонном состоянии (требуется заполнение и проверка).
  • На 01.03.2026 production-нагрузка ограничена; идет подготовка к пилотам/платежам.

3. Техническая архитектура и безопасность

3.1. Production-архитектура

Примечание: текущая версия (MVP) работает как монолит на одном сервере (:3000).
Production-архитектура ниже - целевое состояние (Target Architecture) для масштабирования и enterprise-нагрузок. 

[CDN для статики (Cloudflare / Fastly)]
        |
        v
 [Балансировщик / API Gateway (Kong / Traefik / NGINX)]
        |
        v
 [Сервер(ы) приложений (Node.js/Python/Go)]
        |
        v
 (опционально) Очереди (RabbitMQ / Redis)
        |
        v
 (опционально) Кеш (Redis)
        |
        v
 [База данных (PostgreSQL + модель multi-tenancy)]

API Gateway берет на себя

  • Rate limiting (на уровне IP, API-ключа, пользователя)
  • Аутентификацию и валидацию JWT
  • Маршрутизацию к разным сервисам
  • Сбор метрик (Prometheus)
  • Логирование запросов

3.2. Multi-tenancy: изоляция данных

Выбранный подход

  • Отдельные схемы в одной БД (PostgreSQL)
  • Каждый тенант получает свою схему (например, tenant_123)
  • При подключении middleware устанавливает search_path на схему текущего тенанта
  • Таблицы с одинаковой структурой во всех схемах
  • Суперадмин имеет доступ ко всем схемам через отдельный интерфейс

Пример middleware (Node.js/Sequelize)

// set schema for current tenant
await db.query(`SET search_path TO ${tenantSchema};`);

Ограничения подхода schema-per-tenant

  • Миграции и обслуживание усложняются при росте числа тенантов.
  • Нужна автоматизация миграций и контроль версий схем для каждого тенанта.

План
При масштабировании возможен переход к tenant_id + RLS или hybrid-модели.

3.3. API: версионирование и документация

  • Версионирование через URL: /api/v1/projects, /api/v2/projects
  • OpenAPI (Swagger) спецификация, доступна по /api/docs
  • Политика поддержки старых версий: минимум 6 месяцев после выхода новой
  • Changelog публикуется в документации

3.4. Rate limiting

Эндпоинт Лимит (запросов/мин) Ключ
GET /api/v1/* (публичные) 1000 IP + API-ключ
POST /api/v1/* (создание) 100 API-ключ
/auth/login 5 IP
/auth/register 3 IP
Дашборды/графики 60 Пользователь

Правила обработки 429 (Too Many Requests)

  • Клиент обязан использовать exponential backoff (например: 1s -> 2s -> 4s -> 8s, max 30s).
  • Запрещены бесконечные ретраи без верхнего лимита (max_attempts: {MAX_RETRY_ATTEMPTS}).
  • Для /auth/* и /api/me: при 401/403 ретраи прекращаются до действия пользователя (login).

3.5. Безопасность: чек-лист

  • TLS 1.3 на всех поддоменах, HSTS (max-age=31536000; includeSubDomains; preload)
  • Заголовки безопасности: CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy
  • Защита от брутфорса: задержки, капча после 5 неудачных попыток
  • Аудит действий: логирование всех критических операций в таблицу audit_log
  • Регулярное сканирование уязвимостей: интеграция OWASP ZAP в CI/CD
  • Пентест перед релизом
  • Управление зависимостями: Dependabot, еженедельные обновления
  • Изоляция тенантов на уровне кода: обязательная проверка принадлежности данных

3.7. Мониторинг и оповещение

  • Метрики: Prometheus + Grafana (CPU, RAM, диски, запросы, ошибки, задержки)
  • Логи: ELK или Loki + Grafana
  • Трассировка: Jaeger (если микросервисы)
  • Внешний мониторинг: UptimeRobot / Pingdom (каждые 5 минут из разных регионов)
  • Алерты: PagerDuty / Telegram-бот (критические: недоступность >2 мин, 5xx >5%, диск >80%)

4. Монетизация и биллинг

4.1. Модель монетизации

  • Базовая модель: подписка (SaaS) с помесячной или годовой оплатой.
  • Метрики тарификации: активные пользователи, тенанты, объем API-запросов, интеграции.
  • Дополнительные доходы: Enterprise-поддержка, кастомизация, платные интеграции.

Почему клиенты платят (value drivers)

  • Контроль доступа и безопасность (тенанты/роли/аудит)
  • Надежность интеграций (API-first, версионирование, лимиты)
  • Операционная прозрачность (дашборды, метрики, SLA)

4.3. Выбор платежного провайдера

Рекомендация: начать со Stripe + Stripe Billing (гибкость, прозрачность, EU-поддержка).
По мере роста добавить Paddle как альтернативу для клиентов, которым нужны инвойсы от европейского юрлица.

4.4. Процесс подписки

  • Пользователь нажимает "Начать бесплатный триал" на лендинге.
  • Выбирает тариф (Start/Pro/Business).
  • Вводит email, пароль, название компании.
  • Переходит на страницу оплаты (карта / PayPal / SEPA).
  • После успешного платежа создается аккаунт администратора и первый тенант.
  • Отправляется приветственное письмо с документацией и приглашением на вебинар.
  • Биллинговая система автоматически выставляет счета каждый месяц.

4.5. Партнерская программа

  • Модель: реферер получает 20% от первых 12 месяцев платежей приведенного клиента.
  • Техническая реализация: Stripe Connect или PartnerStack.
  • Юридическое оформление: отдельный договор с партнером (типовой).
